# Formatting Guide — Capacities

คู่มืออ้างอิงสำหรับตัดสินใจว่าควรใช้ formatting แต่ละแบบเมื่อไหร่และเพราะอะไร

---

## ตัวหนา `**text**`

**ความหมาย:** "สิ่งนี้สำคัญ ต้องจำไว้"

ใช้สำหรับ:
- **คำสำคัญ** ที่เป็น core concept ของเนื้อหานั้น เช่น **Machine Learning**, **API Gateway**
- **ชื่อที่ต้องจำ** — ชื่อคน, ชื่อโปรแกรม, ชื่อบริษัท, ชื่อโปรเจกต์
- **ข้อสรุปหลัก** หรือ takeaway ของย่อหน้า
- **ตัวเลขสำคัญ** เช่น เป้าหมาย, งบประมาณ, ขีดจำกัด

อย่าใช้สำหรับ:
- คำทั่วไปที่ไม่ใช่ concept หลัก (จะทำให้ bold เจือจาง)
- ทุกประโยคในย่อหน้า — ถ้า bold ทุกอย่างเท่ากับไม่ได้ bold อะไรเลย

**ตัวอย่าง:**

> ข้อความเดิม: "kubernetes คือระบบจัดการ container ที่ google พัฒนาขึ้น ใช้สำหรับ deploy และ scale แอปพลิเคชัน"
>
> หลัง format: "**Kubernetes** คือระบบจัดการ container ที่ **Google** พัฒนาขึ้น ใช้สำหรับ deploy และ scale แอปพลิเคชัน"

---

## ตัวเอียง `*text*`

**ความหมาย:** "บริบท การอ้างอิง หรือเน้นเบาๆ"

ใช้สำหรับ:
- *ชื่อหนังสือ*, *บทความ*, *เพลง*, *ภาพยนตร์* — สื่อทุกประเภท
- *คำภาษาต่างประเทศ* ที่ยังไม่ได้ทับศัพท์ไทยหรือใช้ทั่วไป
- *นิยามคำแรก* — ครั้งแรกที่นำเสนอคำศัพท์พร้อมอธิบาย
- *การเน้นเบาๆ* ที่ไม่ถึงขั้นต้องตัวหนา เช่น "ควรทำ *ก่อน* เสมอ"

อย่าใช้สำหรับ:
- สิ่งที่ต้องการเน้นแบบ "จำให้ได้" → ใช้ **bold** แทน
- ชื่อ command หรือ technical string → ใช้ `code` แทน

**ตัวอย่าง:**

> "อ่านหนังสือ *Atomic Habits* แล้วนำแนวคิด *habit stacking* มาใช้"

---

## Code `` `text` ``

**ความหมาย:** "ค่าที่ต้องใช้แบบตรงๆ ห้ามเปลี่ยน"

ใช้สำหรับ:
- `คำสั่ง` และ flags เช่น `git commit`, `npm install --save-dev`
- `file path` และ directory เช่น `/etc/nginx/nginx.conf`, `~/Downloads/`
- `ชื่อตัวแปร`, `function`, `class` ในโค้ด
- `ค่า config` หรือ `environment variable` เช่น `NODE_ENV=production`
- `ค่าที่ต้องกรอกแบบตรงๆ` เช่น `true`, `null`, `200`, `utf-8`

สำหรับ code หลายบรรทัด ใช้ code block:

````
```python
def hello():
    return "world"
```
````

**ตัวอย่าง:**

> "รัน `docker compose up -d` แล้วเช็ก log ด้วย `docker logs -f app`"

## Highlight `==text==`

**ความหมาย:** "ห้ามพลาด — หยุดอ่านตรงนี้"

ใน Capacities: แสดงผลเป็น 🟡 **สีเหลือง** — มีแค่สีเดียว ใช้ได้เลยด้วย `==text==`

ใช้สำหรับ:
- ==คำเตือน== หรือ ==ข้อควรระวัง== ที่ถ้าพลาดจะมีผลเสีย
- ==Deadline== หรือ ==วันที่สำคัญ== ที่ต้องจำ
- ==Action item== ที่ต้องทำก่อนอื่น
- ==ข้อสรุปที่สำคัญที่สุด== ของ note ทั้งหมด

อย่าใช้สำหรับ:
- สิ่งที่ "ค่อนข้างสำคัญ" — highlight ควรสงวนไว้สำหรับ "สำคัญที่สุด" เท่านั้น
- ข้อมูลทั่วไปหรือพื้นหลัง

**ตัวอย่าง:**

> "ต้องส่งรายงานภายใน ==วันศุกร์นี้ 17:00== มิฉะนั้นโปรเจกต์จะล่าช้า"

**Frequency สูงสุด:** 0–2 ครั้งต่อ note — ถ้าใช้มากกว่านี้จะสูญเสียพลังดึงดูดสายตา

---

## ขีดเส้นใต้ `<u>text</u>`

**ความหมาย:** "คำนี้กำลังถูกนิยาม หรือเป็นชื่อพิเศษที่ต้องสังเกต"

ใช้สำหรับ:
- <u>คำที่กำลังถูกนิยาม</u> ในประโยคนั้นเอง
- <u>ชื่อเฉพาะที่พิเศษ</u> ที่ไม่ใช่ technical term เช่น ชื่อโปรเจกต์, ชื่อทีม, ชื่อโครงการ
- <u>ลิงก์ที่ต้องการเน้น</u> ว่าคลิกได้และสำคัญ

**หมายเหตุ Capacities:** ใช้ `<u>text</u>` (HTML inline) — render ได้ปกติใน Capacities

อย่าใช้สำหรับ:
- สิ่งที่ควรใช้ **bold** แทน (underline ดู "heavy" กว่า ควรใช้น้อยกว่า)
- technical string → ใช้ `code` แทน

**ตัวอย่าง:**

> "<u>Zero-shot learning</u> คือการที่โมเดล AI สามารถทำงานกับ task ที่ไม่เคยเห็นตัวอย่างมาก่อน"

## Quick Decision Table

```
ถามตัวเองว่า...                           → ใช้
─────────────────────────────────────────────────────
มันคือ command, path, หรือ exact value?  → `code`
ห้ามพลาดอย่างแท้จริง?                    → ==highlight==
มันคือชื่อสื่อหรือคำต่างชาติ?            → *italic*
มันคือ concept หลักที่ต้องจำ?            → **bold**
มันคือคำที่กำลังถูกนิยาม?               → <u>underline</u>
อื่นๆ                                    → plain text
```

---

## สัญญาณเตือนว่า Format มากเกินไป

- Bold มากกว่า 5 คำในย่อหน้าเดียว
- Highlight มากกว่า 2 ครั้งใน note เดียว
- ใช้หลาย formatting บนคำเดียวกัน (เช่น **`code`** หรือ ***bold italic***)
- ทุกประโยคมี formatting — ถ้าทุกอย่างเด่น ไม่มีอะไรเด่นจริงๆ
